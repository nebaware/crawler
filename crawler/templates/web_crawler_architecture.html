‚ö° Concurrency & Multithreading

Why Concurrency Matters

Traditional sequential crawlers process one URL at a time. With network I/O being the bottleneck, this is extremely inefficient.

Sequential Approach:
1000 URLs √ó 2 seconds = 33 minutes
Concurrent Approach:
1000 URLs √∑ 10 workers = 3.3 minutes

Our Implementation

  • Multiple Celery Workers
    Each worker processes tasks independently
  • Task Queue (Redis)
    Distributes work across available workers
  • Async Processing
    Non-blocking I/O operations
  • Horizontal Scaling
    Add more workers on demand

üíª Core Implementation

Celery Task (crawler/tasks.py)

from celery import shared_task
import requests
from bs4 import BeautifulSoup

@shared_task
def crawl_page_task(url):
    # Fetch the page
    response = requests.get(url, timeout=10)
    
    # Parse HTML
    soup = BeautifulSoup(response.text, 'html.parser')
    title = soup.title.string
    
    # Save to database
    CrawledPage.objects.update_or_create(
        url=url,
        defaults={'title': title, 'content': soup.get_text()}
    )

class CrawledPage(models.Model):
    url = models.URLField(unique=True)
    title = models.CharField(max_length=1000)
    content = models.TextField()
    status_code = models.IntegerField()
    crawled_at = models.DateTimeField()
    search_vector = SearchVectorField()
    
    class Meta:
        indexes = [
            GinIndex(fields=['search_vector'])

üîç PostgreSQL Full-Text Search

‚ùå Traditional Search (SLOW)

SELECT * FROM pages
WHERE content LIKE '%python%';

-- Full table scan
-- No ranking
-- Case sensitive
-- ‚è±Ô∏è 5000ms for 1M rows

‚úÖ Full-Text Search (FAST)

SELECT * FROM pages
WHERE search_vector @@ to_tsquery('python')
ORDER BY ts_rank(search_vector, query);

-- GIN index scan
-- Relevance ranking
-- Stemming support
-- ‚ö° 50ms for 1M rows
